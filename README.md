# HoneyBadger
현업 그룹웨어 사용 경험을 바탕으로  
결재·일정·메일·채팅 등 **업무 흐름 중심 협업 기능을 구현한 팀 프로젝트**

---

## 프로젝트 배경

- Java 기반 웹 개발 역량을 학습하기 위한 6개월 국비지원 과정에서 진행한 팀 프로젝트입니다.
- 다만 학습 목적에 그치지 않고,  
  **이전 직장에서 실제 사용하던 그룹웨어의 업무 흐름과 운영 경험을 설계에 적극 반영**했습니다.
- 단순 CRUD 구현이 아니라  
  “왜 이 기능이 필요한가”, “업무에서는 어떤 순서로 사용되는가”를 기준으로 기능을 정의했습니다.
- 결재, 일정, 메일, 채팅 등 주요 기능은  
  **ERP·그룹웨어 운영 경험을 바탕으로 사용자 시나리오 중심으로 구성**했습니다.

---

## 나의 역할 (남도원)

- ERP·그룹웨어 운영 경험을 바탕으로 **업무 흐름 기준 기능 설계 담당**
- ERP·그룹웨어 운영 경험을 바탕으로 업무 흐름 기준 기능 설계 담당
- 전자결재: Back-End 구현 (결재 상태 전이 로직, 예외 케이스 처리)
- 채팅: Front-End 구현

---

## 전자결재 설계 포인트

- 실제 그룹웨어 결재 프로세스를 기준으로 설계
  - 첫 승인자만 `결재 중`
  - 이후 승인자는 `결재 대기`
- 반려 발생 시
  - 이후 승인자 결재 불가 처리
  - 기안서 상태 즉시 `반려`로 변경

**→ 설계 의도:**
  - 실무에서 자주 발생하는 결재 병목과 혼선 최소화
  - 운영 중 문의·오류 발생 시 원인 추적 용이성 확보

---

## 사용 기술

- Back-End: Java, Spring Boot
- Front-End: Next.js
- DB: MySQL
- Infra: Docker, GitHub Actions

※ 운영 환경 분리를 고려하여 Docker 기반 배포 구성

---

## 이 프로젝트를 통해 얻은 관점

- 협업 서비스는 “기능의 많고 적음”보다 **업무 흐름의 일관성**이 중요하다는 점
- 사용자는 기능 자체보다 **프로세스가 어긋날 때** 가장 큰 불편을 느낀다는 점
- 운영자는
  - 사용자 문의를 구조적으로 정리하고
  - 개발자가 이해할 수 있는 형태로 전달해야 한다는 점

→ ERP·협업 서비스 운영 직무를 목표로 하게 된 계기
